# Skill 文档规范

## 1. 概述

本规范定义了 nndeploy 项目 Skill 文档的标准化格式，旨在让 AI Agent 更高效、自动化地调用和理解文档内容，提升编程协作能力。

## 2. Skill 分类体系

### 2.1 按功能分类

| 分类 | 描述 | 目录 |
|------|------|------|
| **Feature** | 新功能设计和实现 | `feature/` |
| **Fix** | Bug 修复和技术改进 | `fix/` |
| **Build** | 构建系统和编译配置 | `build/` |
| **Deploy** | 部署和运维 | `deploy/` |
| **Test** | 测试和验证 | `test/` |
| **Plugin** | 插件开发和扩展 | `plugin/` |
| **Workflow** | 工作流和 DAG 操作 | `workflow/` |
| **Tool** | 工具和脚本 | `tool/` |

### 2.2 按难度分类

| 难度 | 描述 |
|------|------|
| **Easy** | 简单任务，1-2小时可完成 |
| **Medium** | 中等复杂度，半天可完成 |
| **Hard** | 复杂任务，需要1-2天 |
| **Expert** | 专家级任务，需要深入理解架构 |

### 2.3 按优先级分类

| 优先级 | 描述 |
|--------|------|
| **P0** | 紧急，阻塞功能使用 |
| **P1** | 重要，影响用户体验 |
| **P2** | 一般，性能优化或改进 |
| **P3** | 可选，锦上添花 |

## 3. YAML 元数据规范

每个 Skill 文档顶部必须包含 YAML 元数据块：

```yaml
---
name: skill_name                    # 技能名称（唯一标识）
title: 技能标题                       # 中文标题
description: 简短描述技能的功能和用途
category: [feature, fix, build]     # 分类（可选多个）
difficulty: easy|medium|hard|expert  # 难度级别
priority: P0|P1|P2|P3                # 优先级
status: draft|planned|in_progress|completed|deprecated  # 状态
author: 作者姓名                      # 作者
created: 2026-01-01                  # 创建日期
updated: 2026-01-01                  # 更新日期
version: 1.0.0                       # 文档版本
tags: [tag1, tag2]                   # 标签（用于搜索）
depends_on: [skill1, skill2]         # 依赖的其他技能
blocked_by: [issue1, issue2]         # 被哪些问题阻塞
related_to: [skill1, skill2]         # 相关技能
estimated_time: 4h                   # 预估耗时（单位：h/d）
files_affected: [path1, path2]      # 涉及的文件路径
---
```

### 3.1 字段说明

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 技能唯一标识，使用 snake_case |
| `title` | string | 是 | 中文标题，简洁明了 |
| `description` | string | 是 | 一句话描述技能功能 |
| `category` | list | 否 | 分类列表，默认根据目录推断 |
| `difficulty` | enum | 否 | 难度，默认 medium |
| `priority` | enum | 否 | 优先级，默认 P2 |
| `status` | enum | 否 | 状态，默认 planned |
| `author` | string | 否 | 作者 |
| `created` | date | 否 | 创建日期 |
| `updated` | date | 否 | 更新日期 |
| `version` | string | 否 | 文档版本号 |
| `tags` | list | 否 | 标签列表 |
| `depends_on` | list | 否 | 依赖的其他技能 |
| `blocked_by` | list | 否 | 被阻塞的 Issue |
| `related_to` | list | 否 | 相关技能 |
| `estimated_time` | string | 否 | 预估耗时，如 4h, 1d |
| `files_affected` | list | 否 | 涉及文件路径列表 |

### 3.2 Status 说明

| 状态 | 说明 |
|------|------|
| `draft` | 草稿，未完成 |
| `planned` | 已规划，等待实现 |
| `in_progress` | 进行中 |
| `completed` | 已完成 |
| `deprecated` | 已废弃 |

## 4. 文档结构规范

### 4.1 通用结构

```markdown
# [Category]: [Title]

## 概述

简要描述此技能的目的和价值。

## 前置条件

- 必需的知识和工具
- 需要完成的先决任务

## 输入

描述此技能需要的输入信息。

## 输出

描述此技能产生的输出结果。

## 使用场景

列出适用此技能的场景。

## 实施步骤

### Step 1: [步骤名称]

具体操作步骤...

### Step 2: [步骤名称]

具体操作步骤...

## 验收标准

- 验收标准 1
- 验收标准 2

## 注意事项

- 注意点 1
- 注意点 2

## 相关资源

- 相关文档链接
- 参考资料
```

## 5. 命名规范

### 5.1 文件命名

| 类型 | 命名格式 | 示例 |
|------|----------|------|
| Feature | `feature_{name}.md` | `feature_dynamic_graph.md` |
| Fix | `fix_{issue}.md` | `fix_buffer_assignment.md` |
| Build | `build_{target}.md` | `build_cuda.md` |
| Test | `test_{module}.md` | `test_tensor.md` |
| Plugin | `plugin_{name}.md` | `plugin_yolo.md` |

### 5.2 技能名称

- 使用小写字母和下划线
- 使用描述性名称
- 避免使用缩写

## 7. 最佳实践

### 7.1 编写建议

1. **简洁明了**：使用简洁的语言，避免冗长描述
2. **结构清晰**：使用标题层次，保持文档结构清晰
3. **代码示例**：提供可运行的代码示例
4. **测试验证**：每个技能都应包含验收标准
5. **持续更新**：随着代码变更同步更新文档

### 7.2 AI Agent 优化

1. **明确输入输出**：清晰定义技能的输入和输出
2. **步骤清晰**：将复杂任务分解为可执行的步骤
3. **依赖明确**：明确标注依赖关系
4. **验证标准**：提供明确的验收标准
5. **错误处理**：说明常见错误和解决方案
